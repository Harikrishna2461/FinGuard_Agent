1. PRODUCT OVERVIEW
================================================================================

1.1 What is FinGuard Agent?

FinGuard Agent is an artificial intelligence-powered financial portfolio management system designed to monitor user investments, detect financial risks, analyze market trends, and provide intelligent investment recommendations.

1.2 Primary Objectives

- Monitor and manage user stock portfolios
- Detect fraudulent activities and risky transactions
- Analyze market trends and provide sentiment analysis
- Ensure compliance with regulatory requirements
- Deliver intelligent investment recommendations

1.3 Target Users

- Individual investors managing their own portfolios
- Financial advisors providing client guidance
- Compliance officers monitoring regulatory adherence
- Portfolio managers analyzing multi-asset allocations

1.4 Core Features

Feature                          Description
================================================================================
Real-time Portfolio Tracking     View aggregated portfolio value and performance
                                 metrics with up-to-date asset pricing

Risk Detection System            Automatically identify fraudulent patterns and
                                 suspicious transactions using ML algorithms

AI-Powered Analysis              Nine specialized AI agents analyze different
                                 aspects of portfolio management

Market Intelligence              Stock sentiment analysis and identification of
                                 emerging market trends

Compliance Verification          Automatic checking against tax laws and
                                 regulatory requirements

User Interface                   Modern web-based dashboard with charts,
                                 visualizations, and real-time updates

3. SYSTEM ARCHITECTURE
================================================================================

3.1 High-Level Architecture

The system follows a three-tier architecture:

Presentation Layer (Frontend)
- React-based user interface
- Responsive web design
- Real-time data visualization

Application Layer (Backend)
- Flask API server
- Business logic and orchestration
- AI agent coordination

Data Layer
- SQLite database for relational data
- ChromaDB for vector embeddings
- Groq API for LLM inference

3.2 Component Communication Flow

User Action in Browser
  |
  | HTTP Request
  v
React Component -> Flask API Route
  |
  | Route Processing
  v
Application Logic Layer
  |
  | Decision Point
  ├─> Database Query (SQLAlchemy)
  ├─> AI Agent Invocation (CrewAI)
  ├─> ML Risk Scoring (scikit-learn)
  └─> API Calls (Groq)
  |
  | Response Generation
  v
JSON Response
  |
  | HTTP Response
  v
Frontend Update and Render


3.3 Data Processing Pipeline

1. Data Ingestion
   - Receive portfolio data, transactions, market data from frontend
   - Validate input parameters and data types
   - Log all incoming requests

2. Processing Layer
   - Route to appropriate handler (ML scoring, AI analysis, database update)
   - Execute parallel operations where possible
   - Manage concurrent requests

3. Analysis Phase
   - Run machine learning models for risk scoring
   - Execute AI agents for intelligent analysis
   - Aggregate results from multiple sources

4. Response Generation
   - Format results for frontend consumption
   - Include metadata and timestamps
   - Handle error conditions gracefully

5. Frontend Rendering
   - Update React components with new data
   - Re-render UI elements
   - Display visualizations and alerts

4. AI AGENTS
================================================================================

The FinGuard system includes nine specialized AI agents, each with distinct responsibilities. All agents inherit from FinancialBaseAgent and use Groq API for language model inference.

4.1 Portfolio Analysis Agent

Purpose: Assess portfolio composition, diversification, and allocation strategy

Responsibilities:
- Calculate asset allocation percentages by sector and asset class
- Compute diversification metrics and scores (0-100 scale)
- Evaluate portfolio concentration risks
- Analyze historical performance trends
- Generate rebalancing recommendations

Processing Method:
- Groq LLM receives portfolio data and analysis requirements
- Agent performs calculations on allocation percentages
- Generates structured recommendations with rationale

Input Data Structure:
{
  "holdings": [
    {
      "symbol": "AAPL",
      "quantity": 100,
      "current_price": 180,
      "sector": "Technology"
    }
  ],
  "total_value": 500000,
  "target_allocation": {...}
}

Output Data Structure:
{
  "diversification_score": 75,
  "sector_allocation": {
    "Technology": "40%",
    "Finance": "30%",
    "Healthcare": "20%",
    "Other": "10%"
  },
  "concentration_risk": "Medium",
  "recommendations": [
    "Reduce Technology sector exposure",
    "Increase Healthcare holdings"
  ]
}


4.2 Risk Detection Agent

Purpose: Identify fraudulent patterns, anomalies, and financial risks

Responsibilities:
- Screen transactions for fraud indicators using ML models
- Detect unusual trading patterns and behavioral anomalies
- Assess market-based risks and volatility
- Analyze concentration risks across holdings
- Flag compliance violations

Processing Method:
Two-stage approach:
Stage 1 - Machine Learning Pre-Screening
- Rules-based scoring: Apply hardcoded financial rules
- Gradient Boosting: Predict fraud probability from trained model
- Isolation Forest: Identify statistical anomalies

Stage 2 - LLM Expert Analysis
- Pass ML results and transaction context to Groq
- Provide expert interpretation and contextual analysis
- Generate final risk assessment

Input Data Structure:
{
  "transaction_history": [
    {
      "symbol": "AAPL",
      "quantity": 1000,
      "price": 180,
      "type": "BUY",
      "timestamp": "2026-04-22T10:00:00"
    }
  ],
  "portfolio_data": {...}
}

Output Data Structure:
{
  "final_score": 85,
  "risk_label": "HIGH",
  "method": "ensemble",
  "flags": [
    "large_quantity",
    "unusual_timing",
    "concentration_increase"
  ],
  "expert_analysis": "Large purchase detected..."
}

Purpose: Analyze market sentiment and provide trend-based recommendations

Responsibilities:
- Evaluate stock sentiment from multiple sources (bullish/bearish)

- Analyze news sentiment and market events

4.4 Compliance Agent

Purpose: Ensure adherence to regulatory and tax requirements

Responsibilities:
- Review transactions against Pattern Day Trader rules
- Identify wash sale violations
- Screen for Anti-Money Laundering concerns
- Detect insider trading indicators
- Generate tax reports and optimization suggestions
- Track regulatory reporting requirements

Processing Method:
- Rule-based compliance checking against known regulations
- Groq LLM provides expert regulatory interpretation
- Tax calculations based on transaction history

Output Structure:
{
  "violations": [
    {
      "type": "wash_sale",
      "severity": "high",
      "description": "Sale followed by repurchase within 30 days",
      "transaction_ids": [123, 124]

  "tax_recommendations": [
    "Consider tax-loss harvesting",
    "Review long-term holding strategies"
  ]
}


4.5 Alert Intake Agent

Purpose: Receive, categorize, and prioritize alerts from all sources

Responsibilities:
- Process alerts from transactions, portfolio changes, and market events
- Categorize alerts by type and urgency
- Assign priority levels (Critical, High, Medium, Low)
- Enrich alerts with contextual information
- Remove duplicate or redundant alerts
- Route alerts to appropriate specialists

Processing Method:
- Groq LLM evaluates alert significance
- Automatic priority assignment based on risk factors
- Alert aggregation and deduplication

Alert Types:
- Price Alerts: Stock price crosses specified threshold
- Performance Alerts: Portfolio returns exceed thresholds
- Volatility Alerts: Trading volume or price movements spike
- Fraud Alerts: Risk engine detects suspicious activity


4.6 Customer Context Agent

Purpose: Maintain and provide relevant customer information for personalized analysis

Responsibilities:
- Build and maintain customer profile information

- Assess current customer financial needs

Processing Method:

4.7 Explanation Agent

Purpose: Translate complex AI analysis into accessible language for different audiences

Responsibilities:
- Explain AI-generated recommendations in clear language
- Tailor explanations to audience type (customer, advisor, compliance, executive)
- Simplify technical findings without losing accuracy
- Provide reasoning behind recommendations
- Generate compliance-friendly documentation

Processing Method:
- Groq LLM adapts explanation complexity
- Audience-specific terminology and terminology
- Balance between transparency and simplicity

Output Example:
Customer Audience:
"We recommend holding your Apple stock based on strong earnings
performance and positive market sentiment. The stock shows upward
momentum with moderate volatility, suitable for your investment profile."

Compliance Audience:
"Position holds no regulatory violations. Concentration in Technology
sector is within acceptable limits (40% vs 50% maximum). No wash sale
or PDT rule violations detected in transaction history."


4.8 Risk Assessment Agent

Purpose: Perform comprehensive risk evaluation across multiple dimensions

Responsibilities:
- Calculate market risk metrics (beta, correlation)
- Evaluate portfolio volatility and Value at Risk
- Assess liquidity risk across holdings
- Analyze credit risk for bond holdings
- Identify systemic risk exposures
- Generate risk scorecards

Processing Method:
- Quantitative risk calculations
- Groq LLM contextualizes results
- Multi-factor risk scoring

Risk Dimensions Evaluated:
- Market Risk: Exposure to market movements
- Concentration Risk: Over-exposure to single stocks/sectors
- Liquidity Risk: Ability to exit positions quickly
- Credit Risk: Counterparty default risk
- Regulatory Risk: Compliance violation exposure


4.9 Escalation and Case Summary Agent

Purpose: Identify critical issues, manage escalations, and prepare handoff documentation

Responsibilities:
- Evaluate incidents for escalation necessity
- Determine appropriate escalation level and team
- Generate comprehensive case summaries
- Prepare escalation packages with context
- Track and document escalation outcomes
- Identify patterns for process improvement

Processing Method:
- Groq LLM evaluates escalation criteria
- Systematic case documentation
- Automated handoff package generation

Escalation Criteria:
- Fraud detection above confidence threshold
- Regulatory compliance violations
- Unusual or complex situations
- Customer complaints or disputes
- System errors or anomalies

Output Structure:
{
  "escalation_level": "critical",
  "target_team": "compliance",
  "case_id": "CASE_2026_04_022_001",
  "summary": "Large transaction detected...",
  "required_actions": [
    "Verify customer identity",
    "Block transaction pending review"
  ],
  "deadline": "2026-04-22T16:00:00"
}

5.3 Analytics Page

Purpose: Provide advanced analysis tools and AI-generated insights

Sub-sections:

Transaction Risk Analysis
This tool analyzes individual transactions for risk.

Process:
1. User enters transaction details
   - Stock symbol (from dropdown)
   - Transaction type (BUY, SELL, DIVIDEND)
   - Quantity (numeric)
   - Price per share (decimal)
   - Transaction fees (optional)

2. System processes the transaction
   - ML Risk Scoring (< 0.5 seconds)
     Rules-based scoring
     Gradient Boosting model prediction
     Isolation Forest anomaly detection
   - Risk Score Generated (0-100 scale)
   - Risk Label Assigned (LOW, MEDIUM, HIGH)
   - Risk Flags Identified

3. AI Analysis Phase
   - Groq LLM analyzes ML results with context
   - Explains risk factors
   - Provides recommendations

4. Results Display
   Risk Score: 75/100
   Risk Level: MEDIUM
   Confidence: 92%
   
   Risk Factors:
   - Large transaction quantity relative to average
   - Time of day matches fraud patterns
   - Price volatility elevated
   
   Recommendations:
   - Verify transaction details
   - Consider splitting into smaller orders
   - Monitor account for related activity

Quick Recommendation Tool
This provides investment recommendations for specific stocks.

Process:
1. Portfolio Selection
   User selects portfolio from dropdown
   On change: loadPortfolioSymbols() function executes
   API Call: GET /api/portfolio/<id>/assets
   
2. Symbol Filtering
   Symbol dropdown populates with portfolio holdings only
   User cannot recommend stocks not in portfolio
   Filters applied to sidebar dropdowns

3. Risk Profile Selection
   Options: Conservative, Moderate, Aggressive
   Affects recommendation weighting

4. Analysis
   Groq LLM analyzes:
   - Current stock price
   - Technical indicators
   - Market sentiment
   - Portfolio fit and diversification
   - Risk compatibility with profile
   - Customer history and preferences

5. Recommendation Output
   Action: BUY, SELL, or HOLD
   Confidence Level: 0-100%
   Key Reasons: 3-5 main factors
   Risk Assessment: Expected volatility
   Time Horizon: Short, Medium, or Long term

   Summary Section:
   - Overall market sentiment (Bullish/Neutral/Bearish)

   - Number of positive vs negative signals
   - Recommendation: BUY, HOLD, or SELL
   
   Individual Stock Analysis:
   For each selected stock:
   - Stock symbol and name
   - Sentiment: BULLISH/NEUTRAL/BEARISH
   - Confidence score: 0-100
   - Trend direction: Upward/Sideways/Downward
   - Key events and news
   - Technical indicators
   - Recent price action
   
   AI Recommendations:
   - Portfolio-level recommendations
   - Sector rotation opportunities
   - Specific entry/exit points
   - Risk warnings
   - Timing considerations

Data Source:
GET /api/symbols (for symbol list)
GET /api/sentiment?symbols=... (for analysis)

6. API ENDPOINTS
================================================================================

The FinGuard backend exposes 22 RESTful API endpoints for frontend consumption. All endpoints require HTTP requests and return JSON responses.

6.1 Portfolio Management Endpoints

ENDPOINT: POST /api/portfolio
METHOD: POST
PURPOSE: Create a new portfolio
AUTHENTICATION: Required
RATE LIMIT: 10 requests per minute per user

REQUEST BODY:
{
  "user_id": "user_123",
  "name": "My Investment Portfolio",
  "initial_investment": 100000
}

RESPONSE (201 Created):
{
  "id": 1,
  "user_id": "user_123",
  "name": "My Investment Portfolio",
  "total_value": 100000,
  "cash_balance": 100000,
  "created_at": "2026-04-22T10:00:00Z",
  "updated_at": "2026-04-22T10:00:00Z"
}

ERROR RESPONSES:
- 400 Bad Request: Missing required fields
- 401 Unauthorized: User not authenticated
- 409 Conflict: User already has portfolio with same name


ENDPOINT: GET /api/portfolios
METHOD: GET
PURPOSE: List all portfolios for authenticated user
AUTHENTICATION: Required

QUERY PARAMETERS:
- sort_by: created_at, name, total_value (default: created_at)
- order: asc, desc (default: desc)
- limit: 1-100 (default: 50)
- offset: 0+ (default: 0)

RESPONSE (200 OK):
{
  "portfolios": [
    {
      "id": 1,
      "user_id": "user_123",
      "name": "My Investment Portfolio",
      "total_value": 110000,
      "cash_balance": 5000,
      "assets_count": 5,
      "created_at": "2026-04-22T10:00:00Z",
      "updated_at": "2026-04-22T14:30:00Z"
    }
  ],
  "total": 1,
  "limit": 50,
  "offset": 0
}


ENDPOINT: GET /api/portfolio/<id>
METHOD: GET
PURPOSE: Retrieve details for specific portfolio
AUTHENTICATION: Required

PATH PARAMETERS:
- id: Portfolio ID (integer)

RESPONSE (200 OK):
{
  "id": 1,
  "user_id": "user_123",
  "name": "My Investment Portfolio",
  "total_value": 110000,
  "cash_balance": 5000,
  "assets_count": 5,
  "created_at": "2026-04-22T10:00:00Z",
  "updated_at": "2026-04-22T14:30:00Z"

- 404 Not Found: Portfolio does not exist

9. DATABASE MODELS
================================================================================

9.1 Portfolio Model

Table: portfolios

Columns:
  - id (Integer, Primary Key)
  - user_id (String, Unique, Foreign Key)
  - name (String): Portfolio name
  - total_value (Float): Current total value in dollars
  - cash_balance (Float): Available cash
  - created_at (DateTime): Portfolio creation timestamp
  - updated_at (DateTime): Last modification timestamp

Relationships:
  - One Portfolio has many Assets
  - One Portfolio has many Transactions
  - One Portfolio has many Alerts


9.2 Asset Model

Table: assets

Columns:
  - id (Integer, Primary Key)
  - portfolio_id (Integer, Foreign Key to portfolios)
  - symbol (String): Stock ticker symbol
  - name (String): Company name
  - quantity (Float): Number of shares held
  - purchase_price (Float): Average purchase price
  - current_price (Float): Most recent market price
  - asset_type (String): "stock", "etf", "bond", "crypto"
  - sector (String): Industry sector
  - created_at (DateTime)
  - updated_at (DateTime)

Relationships:
  - Many Assets belong to one Portfolio


9.3 Transaction Model

Table: transactions

Columns:
  - id (Integer, Primary Key)
  - portfolio_id (Integer, Foreign Key)
  - symbol (String): Stock symbol
  - transaction_type (String): "buy", "sell", "dividend"
  - quantity (Float): Number of shares
  - price (Float): Price per share
  - total_amount (Float): Quantity * Price
  - fees (Float): Transaction fees
  - risk_score (Integer): ML risk score (0-100)
  - risk_label (String): "LOW", "MEDIUM", "HIGH"
  - risk_flags (JSON): Array of risk indicators
  - created_at (DateTime)
  - timestamp (DateTime): Transaction execution time

Relationships:
  - Many Transactions belong to one Portfolio


9.4 Alert Model

Table: alerts

Columns:
  - id (Integer, Primary Key)
  - portfolio_id (Integer, Foreign Key)
  - symbol (String): Stock symbol
  - alert_type (String): "price", "performance", "volatility", "fraud"
  - condition (String): "above", "below", "equals"
  - threshold_value (Float): Trigger value
  - current_value (Float): Last checked value
  - status (String): "active", "triggered", "completed"
  - is_active (Boolean): Active/inactive flag
  - triggered_at (DateTime): When alert was triggered
  - created_at (DateTime)
  - expires_at (DateTime): Optional expiration date

Relationships:
  - Many Alerts belong to one Portfolio


9.5 Risk Assessment Model

Table: risk_assessments

Columns:
  - id (Integer, Primary Key)
  - portfolio_id (Integer, Foreign Key)
  - market_risk_score (Integer): 0-100
  - concentration_risk_score (Integer): 0-100
  - liquidity_risk_score (Integer): 0-100
  - credit_risk_score (Integer): 0-100
  - systemic_risk_score (Integer): 0-100
  - overall_risk_score (Integer): 0-100
  - risk_level (String): "LOW", "MEDIUM", "HIGH"
  - created_at (DateTime)
  - analysis_details (JSON): Detailed risk breakdown

Relationships:
  - Many Risk Assessments belong to one Portfolio


9.6 Audit Log Model

Table: audit_logs

Columns:
  - id (Integer, Primary Key)
  - portfolio_id (Integer, Foreign Key)
  - user_id (String)
  - action (String): Action type ("transaction", "analysis", "alert")
  - action_details (JSON): Full action details
  - results (JSON): Action results
  - changes_made (JSON): Database changes
  - created_at (DateTime)
  - execution_time_ms (Integer): Processing time

Purposes:
  - Compliance tracking
  - User activity logging
  - System audit trail
  - Performance monitoring

10. IMPLEMENTATION GUIDE

10.1 Backend Setup

Prerequisites:
- Python 3.9+
- pip package manager
- SQLite3
- Environment variables configured

Installation Steps:

1. Install Dependencies
   cd backend
   pip install -r requirements.txt

2. Configure Environment
   Create .env file:
   GROQ_API_KEY=your_api_key_here
   GROQ_MODEL=llama-3.3-70b
   FLASK_ENV=development
   DATABASE_URL=sqlite:///finguard.db
   LOG_LEVEL=INFO

3. Initialize Database
   python -c "from app import db; db.create_all()"

4. Load Knowledge Base
   python load_knowledge_base.py

5. Start Backend Server
   python run.py
   Server will run on http://localhost:5000


10.2 Frontend Setup

Prerequisites:
- Node.js 16+
- npm or yarn
- Modern web browser

Installation Steps:

1. Install Dependencies
   cd frontend
   npm install

2. Configure Environment
   Create .env file:
   REACT_APP_API_URL=http://localhost:5000

3. Start Development Server
